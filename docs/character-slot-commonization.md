# Character Slot Commonization Plan

## 目的
GN-AKARI の scene において、キャラ1とキャラ2で設定・操作・状態保持の差をなくす。

現状はキャラ1が先行実装されており、一部機能がキャラ2に揃っていない。
まずは 2 スロット固定のまま、両者を同一仕様へ揃える。

---

## 今回のゴール
- キャラ1設定と同じことがキャラ2設定でもできる
- キャラ1とキャラ2で UI 上の能力差がない
- キャラ1とキャラ2で state 保存・復元仕様が揃っている
- 将来的なキャラスロット動的追加に進めやすい内部構造に近づける

---

## 今回やること
### 1. source 操作の対称化
- キャラ1とキャラ2の両方で画像選択できる
- portrait / gallery / preview-cache 等の扱いを揃える
- source 切替時の state 更新と preview 反映を揃える

### 2. レイアウト保持の対称化
- X / Y / Scale をキャラ1とキャラ2の両方で画像単位に保持する
- 画像A/Bで別の値を持てる
- 再選択時に復元される
- 再読み込み後も復元される

### 3. ギャラリー適用先の対称化
- ギャラリー画像をキャラ1だけでなくキャラ2にも適用できるようにする
- 適用先は明示的に選択できる形が望ましい

### 4. UI 操作の対称化
- 表示/非表示
- ロック
- レイヤ順参加
- 設定値編集
- Preview反映

これらをキャラ1/キャラ2で揃える。

---

## 今回やらないこと
- キャラ数の動的追加
- キャラスロット追加/削除 UI
- state 保存構造の全面刷新
- scene 全体の大規模再設計

---

## 将来の拡張方針
今回は 2 スロット固定のまま進めるが、将来的にはキャラを固定番号ではなく
「同一仕様の character slot 配列」として扱える構造へ寄せる。

想定イメージ:
- char1 / char2 固有処理を減らす
- slot 引数付き関数へ寄せる
- 保存キーも slot + asset 単位で安定化できるようにする

---

## 実装順
### フェーズ1
- キャラ1/キャラ2 の仕様差を埋める

### フェーズ2
- 内部処理を slot ベースへ寄せる

### フェーズ3
- キャラスロットの動的追加を検討する

---

## 完了条件
- キャラ1でできる操作がキャラ2でもできる
- キャラ2でも画像ごとの X / Y / Scale 保持が動く
- ギャラリー画像をキャラ2にも適用できる
- state 保存/復元でキャラ1とキャラ2の挙動差がない
